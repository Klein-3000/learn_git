# 仓库结构
## 目录结构
```
<BackupRepository>
    ├── config          [加密的仓库元数据]{**二进制**格式，包含仓库ID和加密配置，无法直接阅读}
    ├── data/            [加密的数据块存储]{所有文件内容被分割、去重、压缩并加密后存放于此}
    ├── index/           [加密的索引文件]{映射快照与数据块的关系，加速查找}
    ├── keys/            [密钥文件（JSON）]{包含用于派生主密钥的参数和加密后的密钥数据}
    ├── locks/           [锁文件]{确保仓库操作的原子性和一致性}
    └── snapshots/       [加密的快照元数据]{每个文件记录一次备份的上下文信息，但内容本身是加密的}
```
## 详细说明

### 1. `config`

- **作用**: 这是仓库的“身份证”和“说明书”。
- **内容**: 一个二进制文件，包含了仓库的**唯一 ID**、使用的**加密算法**（如 AES-256）、**压缩设置**等关键元数据。
- **重要性**: 如果 `config` 文件损坏或丢失，整个仓库将无法被 Restic 识别和访问，意味着你的所有备份数据都将无法恢复。
#### 查看 config 文件的内容
```bash
❯ restic cat config
repository 79404b4e opened (version 2, compression level auto)
{
  "version": 2,
  "id": "79404b4ea6056dd180716f4e3701ea536a582f3cc013537ea771b212a2a43d10",
  "chunker_polynomial": "28c27ce8ab8529"
}
```

---

### 2. `data`

- **作用**: 仓库的“心脏”，存放所有实际的备份数据。
- **结构**: 内部包含多层子目录（如 `data/01/`, `data/ab/` 等），这是一种为了在文件系统上高效管理大量小文件而设计的分片策略。
- **内容**: 所有原始文件在备份时会被 Restic **分割成固定大小的数据块（chunks）**，然后经过**去重（deduplication）**、**压缩**和**加密**后，作为独立的文件存储在此处。相同的块只存储一次。
#### 2.1 目录结构
```bash
<BackupRepository>
    └── data            [分片式加密数据存储]{采用256个子目录（00-ff）的哈希分片策略，高效管理海量小文件}
        ├── 00/         [空桶]{仅在仓库初始化时创建，用于保证目录结构完整}
        ├── 01/         [空桶]{同上}
        ...
        ├── 0e/         [活跃数据桶]{LastWriteTime 为 2026/4/7，包含实际的数据块文件}
        ├── 12/         [活跃数据桶]{同上}
        ...
        └── ff/         [空桶]{同上}
```

```powershell
# 列出当前目录下所有非空的子目录
 Get-ChildItem -Directory | Where-Object { (Get-ChildItem $_.FullName -Force | Measure-Object).Count -gt 0 }
```

#### 2.2 固定的 256 个子目录 (`00` - `ff`)

- **现象**: `ls` 命令显示 `data` 目录下总共有 **256** 个子目录，命名从 `00` 到 `ff`。
- **原理**: `[这是基于哈希的分片（sharding）策略]{Restic 会对每个数据块的内容计算一个 SHA-256 哈希值。取该哈希值的前两个十六进制字符（即第一个字节），作为子目录名。}`
    - 例如，一个数据块的哈希是 `a7b3c...`，它就会被存放在 `data/a7/` 目录下。
- **目的**:
    - **性能优化**: 避免在单个目录中存放成千上万个文件，这在许多文件系统（尤其是 NTFS 和 ext4）上会导致严重的性能下降。
    - **可预测性**: 这种静态的、固定的分片方案简单高效，无需动态管理分片元数据。

#### 2.3 “空桶”与“活跃桶”的区分

- **现象**: 您使用 `Get-ChildItem ... | Where-Object { ...Count -gt 0 }` 精准地筛选出了 **34 个**非空的子目录（如 `0e`, `12`, `15` 等），它们的 `LastWriteTime` 都是 `2026/4/7`。而其余 222 个目录虽然存在，但内部为空，其 `LastWriteTime` 是仓库初始化时间 `2026/2/27`。
- **分析**:
    - **空桶 (`00`, `01`, ..., 大部分目录)**: `[仓库初始化时预创建]{为了简化代码逻辑，Restic 在创建仓库时会一次性生成全部 256 个子目录。}` 即使当前没有对应哈希前缀的数据块，目录也存在。
    - **活跃桶 (`0e`, `12`, `15`, ...)**: `[真实数据的容器]{这些目录包含了实际的、加密后的数据块文件。它们的修改时间反映了最近一次向其中写入数据的时间（即您最近的备份操作）。}`
#### 2.4 数据块文件本身

- **内容**: 每个非空子目录（如 `0e/`）内部的文件，就是 Restic 的**数据块（chunk）**。
- **特性**:
    - **加密**: 所有数据块在写入磁盘前都已使用仓库密钥进行了**强加密**，无法直接读取。
    - **去重**: 如果两个完全相同的文件（或文件片段）被备份，它们只会产生一个数据块，并被存储一次。这就是 Restic 节省空间的核心机制。
    - **固定大小**: Restic 使用可变大小的分块算法（Rabin Fingerprinting），但块大小通常在一个范围内（默认目标大小为 1MiB），以平衡去重效率和索引开销。

#### 2.5 与其他组件的关联

- **`index` 目录**: 索引文件记录了“哪个快照需要哪些数据块”。当执行 `restic restore` 时，Restic 通过索引找到所需数据块的哈希，然后根据哈希前缀（如 `a7`）定位到 `data/a7/` 目录，最后读取具体的块文件。
- **`snapshots` 目录**: 快照文件定义了备份的目录树结构，而目录树中的每个文件都由一个或多个指向 `data` 目录中数据块的指针组成。


---

### 3. `index`

- **作用**: 仓库的“目录”或“地图”。
- **内容**: 包含了多个索引文件。每个索引文件记录了特定时间段内创建的**快照（snapshot）** 与 `data` 目录中**数据块**之间的映射关系。
- **工作原理**: 当你执行 `restic restore` 或 `restic find` 时，Restic 首先会读取 `index` 文件，快速定位到所需文件由哪些数据块组成，然后再去 `data` 目录中拼凑出完整的文件。没有索引，恢复过程将变得极其缓慢。

---

### 4. `keys`

- **作用**: 仓库的“钥匙串”。
- **内容**: 存放着一个或多个**加密密钥**。每个密钥文件本身也是被你的仓库密码加密的。
- **安全性**: 拥有 `keys` 目录中的任意一个有效密钥文件和对应的密码，就可以完全访问仓库内的所有数据。因此，**保护好你的密码和密钥文件至关重要**。

---

### 5. `locks`

- **作用**: 仓库的“门锁”。
- **内容**: 当一个 Restic 命令（如 `backup`, `forget`）正在运行并需要修改仓库时，它会在此目录下创建一个锁文件。
- **行为**: 如果另一个 Restic 命令尝试同时访问仓库，它会检测到锁文件的存在，并等待或报错（退出状态码 11）。正常操作结束后，锁文件会被自动清理。

---

### 6. `snapshots`

- **作用**: 仓库的“相册”。
- **内容**: 每个快照文件是一个[**二进制**文件]{文件名对应 **snapshot_id** }，记录了该次备份的**元数据**，包括：
    - 备份的时间戳
    - 被备份的主机名 (`hostname`)
    - 被备份的路径列表 (`paths`)
    - 用户添加的标签 (`tags`)
    - 该快照所引用的**树对象（tree）** 的 ID（这个树对象描述了备份时刻的完整目录结构）
- **用途**: `restic snapshots` 命令就是通过读取这个目录下的文件来列出所有可用的快照。
