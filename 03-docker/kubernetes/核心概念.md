# Kubernetes组件
1. Node ：可以是物理机或虚拟机 --> docker? 
	1. [pod]{豆荚} : 不是一个稳定的实例,Kubernetes中最小调度单元 --> container?
	2. [ing]{入口} : 统一的入口
	3. [vol]{volumes} : 数据持久化
2. [cm]{configMap} : 统一管理配置,明文配置,不安全
3. [svc]{service} : 类似域名的作用
4. [secret]{秘密} : base63 后的 cm
5. 部署
	1. [deploy]{deployment} : 无状态
	2. [sts]{statefulset} : 有状态的,如数据库,有多少条数据,已经数据库备份了多少
# Kubernetes 架构
Master(管理) worker(工作)
Node
	kubelet : 管理和维护
	kube-proxy : 网络代理,负载均衡
	container-runtime
Master(control plane)
	api
	[sched]{scheduler} : ==监控==资源状况
	etcd : 键-值存储系统==存储==资源对象的状态
	[c-m]{controller Master:控制器管理器} : ==管理==集群中各种**资源对象**的==状态==
	[c-c-m]{cloud-controller-Master : 云控制器管理器} : 云服务商

| 组件                                                    | 角色                                                  | 是否“干活”                                         |
| ----------------------------------------------------- | --------------------------------------------------- | ---------------------------------------------- |
| `kubectl`                                             | 用户与 Kubernetes 集群交互的命令行工具  <br>（类似 Web UI 的 CLI 版本） | ❌ 不干活  <br>只发请求（如 `create`, `apply`, `delete`） |
| Kubernetes API Server                                 | 集群的“入口”，接收所有请求（来自 `kubectl`、Web UI、Operator 等）      | ❌ 不直接干活  <br>负责认证、鉴权、验证、持久化到 etcd              |
| etcd                                                  | 集群的“数据库”，存储所有资源状态（Pod、Service 等）                    | ❌ 只存数据                                         |
| Controller Manager                                    | 负责“期望状态 vs 实际状态”的 reconcile（如 Deployment 控制器）       | ⚙️ 逻辑控制  <br>但不碰容器                             |
| Scheduler                                             | 决定 Pod 跑在哪个 Node 上                                  | ⚙️ 调度决策  <br>不启动容器                             |
| Kubelet（运行在每个 Node 上）                                 | 关键！ 负责与容器运行时通信，管理 Pod 生命周期                          | ✅ 真正“监工”  <br>调用 containerd/Docker 启停容器        |
| Container Runtime（如 containerd, Docker Engine, CRI-O） | 实际创建、运行、销毁容器的底层引擎                                   | ✅ 真正在干活                                        |

## deployment
```yaml
apiVersion: [apps/v1]{group/version}
[kind]{指定资源类型}: Deployment
metadata:
	name: nginx-deployment
[spec]{specification : 规范}:
	selector:
		matchLabels:
			app: nginx
	replicas: 3
	template:
		metadata:
			labels:
				app: nginx
		spec:
			containers:
			  - name: nginx
			    image: nginx:1.25
			    ports:
					- containerPort:80

 
```

## Service
```yaml
apiVersion: v1
kind: Service
metadata:
	name nginx-service
spec:
	[type: NodePort]{默认为 **ClusterIP**,既 内部服务}
	selector:
		app: nginx
	ports:
	- protocol : TCP
	  [port]{**暴露**的端口} : 80
	  [targetPort]{**pod**的端口}: 80
	  [nodePort: 30080]{范围必须在 **30000~32767**}
```
**Service 的种类**
	- **ClusterIP**      默认类型，集群内部的服务
    - **NodePort**       节点端口类型，将服务公开到集群节点上
    - **LoadBalancer**   负载均衡类型，将服务公开到外部负载均衡器上
    - **ExternalName**   外部名称类型，将服务映射到一个外部域名上
    - **Headless**       无头类型，主要用于DNS解析和服务发现
