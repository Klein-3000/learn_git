```Dockerfile
...
CMD ["/bin/sh"]

```
## 常驻(-dit)
```bash
docker run -dit --name <container-name> <image> sh
```

# 重启策略
| 策略（`--restart` 值）          | 容器退出时的行为                                        | 是否随 Docker 守护进程启动而启动 | 适用场景                                        | 注意事项                                                           |
| -------------------------- | ----------------------------------------------- | -------------------- | ------------------------------------------- | -------------------------------------------------------------- |
| `no`（默认）                   | 从不重启  <br>容器退出后保持停止状态                           | ❌ 否                  | - 调试临时容器  <br>- 一次性任务（如 `docker run --rm`）  | 即使 Docker 服务重启，容器也不会自动启动                                       |
| `always`                   | 总是重启  <br>无论退出码为何（包括 `0`）                       | ✅ 是                  | - 关键后台服务（如数据库、Web 服务器）  <br>- 需要 7×24 运行的应用 | 即使手动执行 `docker stop`，Docker 守护进程重启后仍会启动该容器（⚠️ 可能不符合预期）         |
| `on-failure[:max-retries]` | 仅非 0 退出码时重启  <br>可选最大重试次数（如 `on-failure:5`）     | ✅ 是                  | - 批处理任务（可能因临时错误失败）  <br>- 需要容错但避免无限循环的任务    | - 若未指定 `max-retries`，则无限重试  <br>- 退出码为 `0` 时不重启                |
| `unless-stopped`           | 总是重启，除非被手动停止  <br>（即：非 `docker stop` 导致的退出都会重启） | ✅ 是                  | - 最推荐的生产策略  <br>- Web 服务、API 网关等常驻服务        | - 手动 `docker stop` 后，即使 Docker 守护进程重启，容器也不会自启  <br>- 兼顾可靠性与可控性 |
